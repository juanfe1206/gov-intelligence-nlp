## Challenges and Friction

- Repeated review findings indicate recurring quality themes: fallback correctness, edge-case handling, and subtle contract mismatches between backend and frontend.
- Some implementation details were corrected late in the story lifecycle (double decode, label fallbacks, complexity inefficiencies), suggesting earlier guardrails would reduce rework.
- Type strictness and schema consistency required repeated fixes (`NarrativeCluster`, `QAFilters`, and frontend type assumptions).
- Frontend Q&A component accumulated complexity in a single file, increasing the chance of regressions when adding behavior.
- Several UX polish concerns were intentionally deferred, creating a manageable but explicit carry-over list for Epic 4 readiness.

## Pattern Analysis Across Story Records

### Common Struggles

- **Fallback logic quality** appeared in multiple stories (empty output handling, null/label fallback behavior, empty-state messaging).
- **Contract drift risk** between API payloads and UI assumptions repeated across 3.3-3.6.
- **Resilience handling** improved incrementally but required multiple review rounds to fully harden retry, abort, and stale-state behavior.

### Recurring Review Themes

- Enforce strict handling of edge inputs and low-confidence states:
  - empty or whitespace LLM outputs
  - missing/null optional fields
  - no-result filter combinations
- Keep semantics aligned across stack:
  - grouping keys vs human-readable labels
  - URL param handling and decode behavior
  - response shape parity and type safety
- Prefer complexity-aware implementations in hot paths (avoid avoidable quadratic patterns).

### Breakthrough Moments

- Story 3.1 established reusable retrieval + aggregation primitives and an explicit insufficient-data contract that later stories leveraged cleanly.
- Story 3.5 converted rapid-response workflow from manual reformulation to one-click context handoff (dashboard to Q&A with prefilled query/filter).
- Story 3.6 transformed failure paths into guided recovery behavior, materially improving demo confidence.
