## What went well

- **Reuse of Story 1.6 jobs API:** 4.1 delivered the ops dashboard without new backend surface area for listing jobs or retry; source volumes derived client-side from job payloads.
- **Incremental hardening of `AdminContent`:** Health strip (4.2), then demo config (4.3), then reset UI (4.4) built on one component with consistent fetch, button, and token patterns.
- **Review loops caught high-severity issues early:** Examples include undefined `jobId` in retry error state (4.1), misleading DB status when the API is unreachable (4.2), README and env gaps for classroom networking (4.3), and reset endpoint robustness (`rowcount`, optional body, error handling) (4.4).
- **Demo operator story (4.3) improved operability:** SQL echo gated on `APP_ENV`, `.env.example` completed against `Settings`, README Demo Setup covers CORS, bind addresses, and build-time caveats for `NEXT_PUBLIC_*`.
- **Destructive action safety:** Reset flow used FK-safe delete order in the backend and a two-step confirm pattern in the UI.

## Challenges and friction

- **`AdminContent.tsx` concentration:** Multiple stories extended the same file; Epic 3’s warning about single-file complexity applies here — mitigated by reviews but still a regression hotspot.
- **Semantic health UI:** Distinguishing "database down" vs "cannot reach API" required an explicit `unknown` path (4.2) — easy to get wrong without review.
- **Unauthenticated admin surface:** Intentional for MVP/demo, but reviews repeatedly flagged auth, CSRF, and rate limiting as deferred — acceptable for stated scope, not for production.
- **Process gap — Story 4.4 smoke checklist:** Task-level smoke boxes remain unchecked in the story file while status is `done`; closeout discipline should align checklist completion with status.

## Pattern analysis across story records

### Common struggles (2+ stories)

- **Design token correctness:** `warning` vs `sentiment-warning`, retry button styling parity with `QAContent` (4.1).
- **Cross-layer semantics:** Health fetch behavior, JSON validation, and API-down vs DB-down (4.2).
- **Configuration and documentation drift:** `.env.example` completeness, duplicate keys, network bind guidance (4.3).
- **Destructive operations:** FK order, async `rowcount`, optional JSON body on POST reset (4.4).

### Recurring review themes

- Match **outline button** and **`type="button"`** patterns to existing components; add **aria-labels** on actionable controls where users disambiguate rows.
- **Parse structured errors** from FastAPI (`detail` shapes) consistently.
- **Abort/cancel** in-flight fetches on unmount where the rest of the app does (4.2).
- **Defer with explicit pre-existing label:** auth hardening, pool timeouts in prod, parallel health fetches — captured as debt, not ignored.

### Breakthrough moments

- **4.2:** Clarifying "DB unknown" when the API is unreachable fixed a misleading ops signal.
- **4.3:** Treating demo as a first-class documented path (README + env) reduces classroom failure modes without adding auth complexity.
- **4.4:** Optional body on `POST /admin/reset` improves operator ergonomics for quick `curl` usage.

### Technical debt (documented in stories)
